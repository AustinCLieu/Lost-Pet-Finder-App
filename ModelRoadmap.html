Initialize with Expo + TypeScript
TypeScript is a stricter version of JavaScript that catches errors while you type, before you even run the app. For example, if Supabase returns a pet report object and you try to access a field that doesn't exist, TypeScript warns you immediately. It makes your code more reliable and is the professional standard for React Native projects.
Set up the monorepo folder structure
Your project has three distinct parts — the mobile app, the database config, and the ML server — that each get deployed differently. Keeping them in one root folder (a monorepo) means one Git repo, one place to look, and easy cross-referencing.
Create lost-pet-app/ as your root
This is the single folder you open in VS Code and push to GitHub. Everything lives inside it.
Move your Expo project into mobile/
Naming it mobile/ (not frontend/) is precise — it signals this is a native phone app, not a website.
Create empty supabase/ and ml-service/ folders with a placeholder .gitkeep file
Git doesn't track empty folders. A blank .gitkeep file makes them show up in your repo so the structure is visible from day one.
Create the root .gitignore
This file tells Git what to never commit. Exclude .env files (contain secret keys), node_modules/ (huge, auto-generated), the ML dataset (gigabytes of images), and the trained model weights file.
Create the root README.md
Start it now even if it's mostly empty. Document how to run each part of the project as you build it. Future you — and anyone else looking at your code — will need this.

Set up Supabase (dashboard)
done
▶
Create your tables in the SQL editor
Your database needs a defined shape before your app can store anything. You write SQL once in the Supabase dashboard to create the tables, then your app uses the JS client — never raw SQL again.
Create profiles table
Supabase creates an auth.users table automatically when someone signs up, but it's locked. You create a profiles table to store extra info like display name, linked via the same user ID. Use on delete cascade so the profile is cleaned up if the user deletes their account.
Create pet_reports table
The core table. Use on delete set null on user_id so reports survive even if the submitter deletes their account — the sighting is still useful to someone looking for their lost pet.
Save SQL as files in supabase/migrations/
Even though you ran them in the dashboard, saving them as numbered SQL files gives you a paper trail if you ever need to rebuild.
Create a Storage bucket called pet-photos
Supabase Storage holds the actual image files. Your database table only stores the URL pointing to the image — not the image itself. This keeps the database fast and lean.
Write RLS policies
Because you enabled automatic RLS, all tables are locked by default. You write policies to open up exactly what you need.
"Anyone can read pet_reports"
The whole point of the app is that anyone can browse sightings — even logged-out users.
"Only logged-in users can insert a report"
Prevents anonymous spam. A user must have an account to submit a sighting.
"Users can only delete their own reports"
The policy checks that auth.uid() = user_id before allowing deletion.
Connect to your Expo app
Copy Project URL and anon key into mobile/.env
Use the EXPO_PUBLIC_ prefix — Expo only injects variables into your app if they start with this.
Create mobile/lib/supabase.ts
Creates the Supabase client once and exports it. Every other file imports from here. The AsyncStorage option saves the login token to the device so users stay logged in.
Test with a .select() call
An empty array back with no error means the connection is live.
3
Build authentication
login + signup + profile trigger
▶
Create the profile trigger in Supabase new
Your profiles table exists but nothing auto-creates a row when a user signs up. Without this, signUp() creates an auth account but the profiles table stays empty — and your app will break later when it tries to load a profile that doesn't exist. You need a database trigger that fires automatically every time a new row is inserted into auth.users. It takes the new user's id and inserts a matching row into profiles. This happens at the database level in a single transaction — either both succeed or neither does.
Write the trigger function and trigger in the SQL editor
The function runs with security definer so it has permission to react to inserts on the locked auth.users table. The trigger fires after insert — meaning the auth user is fully created before the profile row is added.
Save as supabase/migrations/004_create_profile_trigger.sql
Keeps your migration trail complete. Anyone rebuilding the database from scratch can run 001 through 004 in order and get the same result.
Test by creating a user in the Supabase Auth dashboard
Go to Auth → Users → Add User. Then check the profiles table — a new row with that user's ID should appear automatically. If it doesn't, the trigger has a bug.
Write Storage RLS policies new
Your pet-photos bucket is locked by default just like your tables. Without RLS policies on Storage, photo uploads will silently fail. You need policies to allow logged-in users to upload images and allow anyone to view them (so photos show up in the public feed).
"Anyone can read files in pet-photos"
Photos need to be publicly viewable — anyone browsing sightings should see the images, even without an account.
"Logged-in users can upload to pet-photos"
Only authenticated users can add images. This prevents anonymous abuse of your storage bucket.
"Users can only delete their own uploads"
Organize uploads by user ID in the path (e.g. user-id/filename.jpg) so the policy can check ownership. Save as supabase/policies/storage_rls.sql.
Create shared TypeScript types new
Before writing any auth or data logic, define the TypeScript types that the rest of your app will use. Create lib/types.ts with types like Profile (matching your profiles table columns: id, display_name, avatar_url, created_at) and PetReport (matching pet_reports: id, user_id, breed, description, photo_url, location_text, latitude, longitude, created_at). Every function and component imports from this one file. If you add a column to the database later, you update the type here and TypeScript will flag every place in your code that needs updating.
Create lib/auth.ts
Contains all your auth logic in one place — signUp(), signIn(), signOut(). Your screens call these functions and don't need to know the Supabase details. Each function is async because it makes a network request to Supabase and has to wait for a response. They all follow the same pattern: call a Supabase auth method, check if an error came back, and either throw that error or return the user. Throwing means your screens can use try/catch — if it throws, show the error; if it doesn't, the login succeeded.
Build (auth)/login.tsx
Collects email and password, calls signIn(), and handles the result. TypeScript helps here because you can type your form state — email: string, password: string, loading: boolean, error: string | null.
Add email + password text inputs
Use secureTextEntry on the password field so it shows dots instead of characters. Set autoCapitalize="none" on email so the keyboard doesn't capitalize the first letter.
Show a loading spinner while the request is in flight
Disable the button while loading === true to prevent double-submits. Use ActivityIndicator from React Native.
Show an error message if credentials are wrong
Supabase returns an error object with a message. Display it below the form in red text. Clear it when the user starts typing again.
Add a "Don't have an account? Sign up" link new
Users need a way to navigate between login and signup. Use Expo Router's Link component or router.push('/(auth)/signup'). Without this, new users are stuck on the login screen with no way to create an account.
Build (auth)/signup.tsx
Same layout as login but with an extra field and a different function call. You can disable email confirmation in the Supabase dashboard under Auth → Settings for easier development — otherwise new users have to check their email before they can log in, which slows down testing.
Add a display name field above email and password new
Your profiles table has a display_name column. If you don't collect it during signup, it stays null and pet report cards will show no submitter name. Collecting it now means the profile is complete from the start.
After signUp() succeeds, update the profile with the display name new
The trigger creates the profile row with only the id. Immediately after signup, run supabase.from('profiles').update({ display_name }).eq('id', user.id) to fill in the name. This is a second network call — wrap it in its own try/catch so a name-save failure doesn't block the user from entering the app.
Add an "Already have an account? Log in" link new
Same navigation pattern as the login screen — users need to be able to go back and forth between the two auth screens.
Create context/AuthContext.tsx
React Context shares data across your whole app without passing it through every component manually. Your AuthContext listens to Supabase's auth state changes using supabase.auth.onAuthStateChange() and stores the current user. Every screen can then check whether a user is logged in by reading from the context, without each screen needing to call Supabase directly. TypeScript lets you define the shape: user: User | null, loading: boolean.
Include a loading state new
When the app first opens, it takes a moment to check if there's a saved session. During that time, user is null — but that doesn't mean the user is logged out, it means you're still checking. Without a loading flag, the auth gate will flash the login screen for a split second before redirecting to tabs. Set loading = true initially, and switch to false once onAuthStateChange fires.
Fetch the user's profile alongside auth state new
When onAuthStateChange fires with a valid session, also fetch the user's profiles row and store it in context. This way every screen has access to display_name and avatar_url without making separate queries. Add a profile: Profile | null field to the context type.
Add auth gate to app/_layout.tsx
The root layout checks: is loading still true? If yes, show a splash screen or spinner. Is user null? Go to (auth)/login. Is user present? Go to (tabs). This one piece of logic protects every tab screen automatically. No individual screen needs to check for authentication — it's all handled here.
Test the full auth flow
Sign up with a display name → check profiles table in Supabase dashboard (should have the name) → land on tabs → force-close and reopen → still logged in (no login flash) → sign out → back at login → tap "Sign up" link → navigate to signup → sign in with existing account → land on tabs. Don't skip this — auth bugs caught now save hours later.

Add an empty state new
When there are zero reports, don't show a blank screen. Show a friendly message like "No sightings yet — be the first to report one!" with a button linking to the Report tab. Use FlatList's ListEmptyComponent prop.
Build (tabs)/sightings.tsx — list view

Build pet/[id].tsx — detail page
The [id] is Expo Router's dynamic route. Tapping a PetCard navigates to /pet/abc123 and this screen calls getReportById(id) to fetch that specific report. Show the full-size photo, breed, description, location text, map snippet, submitter name, and timestamp.

Compare report.user_id with the current user's ID from AuthContext. If they match, show a delete button. If they don't, hide it. RLS will block unauthorized deletions on the backend too, but hiding the button is better UX than showing an error.

Offer both camera and photo library options new
Show two buttons: "Take Photo" (opens camera) and "Choose from Library" (opens gallery). Some users will snap a photo of a pet they just saw; others will upload one they took earlier.
Build components/LocationPicker.tsx
Uses expo-location to get GPS coordinates or let users pin a location on a map. Latitude and longitude get saved to pet_reports and power the map screen later.

Same pattern as camera permissions. Call requestForegroundPermissionsAsync(). If denied, let users type a location name manually into the location_text field as a fallback — without lat/long the pin won't show on the map, but the report is still useful.
Auto-fill with current location, allow manual adjustment
Default to the user's GPS location but show a small map where they can drag the pin. The sighting might not be where the user currently is — they could be reporting from home about a pet they saw at the park.
Build lib/storage.ts

5
Build and train the CNN model
Python + ML
▶
Set up the ml-service/ folder
A completely separate Python project inside your monorepo. Its own dependencies, its own runtime, deployed to a different server than anything else. Create a Python virtual environment here (python -m venv venv) so your ML dependencies don't conflict with anything else on your system. Add venv/ to .gitignore.
Get a dog breed dataset
A CNN learns by seeing thousands of labeled examples. The Stanford Dogs Dataset has 20,580 images across 120 breeds and is free. Download into training/dataset/ — add to .gitignore since it's several gigabytes.
Write training/train.py using transfer learning
Transfer learning shortcuts training from scratch — you take MobileNetV2 or ResNet50 (already trained on 1.2 million images) and add your own classification layer on top for dog breeds. This means you need far fewer images and far less training time than building a CNN from zero.
Split the dataset into training, validation, and test sets new
Use roughly 70% for training, 15% for validation (checked during training to detect overfitting), and 15% for final testing (never seen during training). If you only have training and test, you can't tell if the model is memorizing instead of learning.
Add data augmentation new
Randomly flip, rotate, zoom, and adjust brightness of training images. This artificially increases your dataset size and makes the model robust to photos taken from different angles, lighting conditions, and distances — exactly what real users will submit.
Load MobileNetV2 with include_top=False
Removes the final ImageNet classification layer so you can add your own for 120 dog breeds. The rest of the network — all the layers that detect edges, shapes, textures, and patterns — stays intact.
Freeze the base model layers
Keeps pretrained knowledge intact. You only train your new classification layer — much faster. Once that layer converges, you can optionally unfreeze the last few base layers and fine-tune with a very low learning rate for a small accuracy boost.
Add a Dense output layer with softmax activation
One neuron per breed (120 total). Softmax converts outputs into probabilities summing to 1 — e.g. "72% Labrador, 18% Golden Retriever, 10% other." The highest probability is the prediction; the value itself is the confidence score your app will display.
Train on Google Colab — free GPU
Training on your laptop could take days. Colab gives you a free GPU that reduces this to hours. Upload your dataset to Google Drive, mount it in Colab, run training, and download the .h5 model file when done.
Write training/evaluate.py
Tests accuracy on the held-out test set — photos the model has never seen during training. A good transfer-learning model on dog breeds typically reaches 80–90% accuracy. If accuracy is much lower, common fixes are: more training epochs, unfreezing a few base layers, or more aggressive data augmentation.
Save outputs to model/
Save the model as breed_cnn.h5 and breed names as classes.json. The classes file maps numeric outputs (0, 1, 2…) back to breed names like "Labrador", "Poodle". Both files must travel together — if you retrain the model, you must regenerate classes.json to match. Add breed_cnn.h5 to .gitignore (it's too large for Git) and note in the README where to download it.
6
Build and deploy the ML API server
FastAPI + deployment
▶
Write preprocess.py
Your CNN was trained on images resized to a specific size (224×224 for MobileNetV2) with pixel values normalized to a 0–1 range. Any image sent from the app must be transformed the exact same way — otherwise the model produces garbage output. This file contains one function that takes raw image bytes from the upload, resizes, normalizes, and returns a NumPy array ready for prediction.
Write model.py
Loads breed_cnn.h5 into memory once when the server starts — not on every request, which would add seconds of delay. Exposes a predict(image_bytes) function that runs preprocessing, calls model.predict(), finds the highest-probability class, and returns the breed name and confidence score using classes.json.
Write main.py — the FastAPI server
FastAPI turns your Python functions into API endpoints that your Expo app can call over the internet. One main route — POST /classify — accepts an uploaded image file and returns the breed prediction as JSON like {"breed": "Golden Retriever", "confidence": 0.91}.
Add CORS headers
Without CORS headers, requests from your app get blocked by browser/platform security rules. FastAPI has a one-line middleware (CORSMiddleware) to allow requests from any origin.
Add a GET /health route
Returns {"status": "ok"}. Deployment platforms like Render ping this endpoint to check if your server is alive. Your app can also call it on startup to warn users if the ML service is down.
Validate the uploaded file new
Check that the file is actually an image (JPEG, PNG) and within a reasonable size limit (e.g. 10MB) before passing it to the model. Return a clear error message if validation fails — this prevents crashes from corrupt uploads or someone sending a non-image file.
Add basic logging new
Log each request with a timestamp, file size, predicted breed, and confidence. When something goes wrong in production, logs are the only way to figure out what happened. Python's built-in logging module is enough.
Write requirements.txt
Python's equivalent of package.json. Lists every library your server needs so the deployment platform can install them: fastapi, uvicorn, tensorflow, pillow, python-multipart. Pin versions (e.g. fastapi==0.104.1) so your deployment doesn't break when a library updates.
Write the Dockerfile
Packages your Python server into a self-contained container that includes the OS, Python, all dependencies, the model file, and your code. The deployment platform reads this and knows exactly how to run your server — no manual setup needed. Use python:3.11-slim as the base image to keep the container small.
Test locally with Postman or curl
Run uvicorn main:app --reload and send a test image to http://localhost:8000/classify. Try multiple breeds, a non-dog image (should still return something — check the confidence is low), an invalid file, and a very large image. Fix bugs here — much easier than debugging on a deployed server.
Deploy to Render or Railway
Both read your Dockerfile and host your server in the cloud. Render has a free tier (the server sleeps after 15 minutes of inactivity and takes ~30 seconds to wake up — fine for a school project). Push code to GitHub, connect the repo to Render, and get back a public URL like https://your-breed-api.onrender.com.
Test the deployed endpoint new
After deployment, hit /health to verify it's running, then send a test image to /classify using the public URL. Response times will be slower than local — especially on a free tier. Note the typical response time so you know what to expect in the app.
7
Connect ML to the app + polish
integration + final
▶
Create mobile/lib/breedClassifier.ts
Bridge between the Expo app and your Python server. Takes a photo URI, converts it into a multipart/form-data POST request, sends it to your deployed API, and returns the breed and confidence. TypeScript return type: Promise<{ breed: string, confidence: number }>. This is an async function because it makes a network request to your ML server and has to wait for the response.
Add the ML URL to mobile/.env
Add EXPO_PUBLIC_ML_API_URL=https://your-breed-api.onrender.com. Storing the URL in .env means if you change hosting providers, you change one line in one file — no digging through code.
Wire the classifier into report.tsx
After the user picks a photo, automatically call classifyBreed(photoUri) in the background. Pre-fill the breed text field with the result. The user can accept the prediction or override it by typing a different breed if the model is wrong.
Show a loading indicator on the breed field while classifying
The API call takes 1–3 seconds (longer if the Render free tier is waking up). Show a spinner on the breed field specifically — don't block the whole form. The user can fill in the description and location while the classifier runs.
Build components/BreedBadge.tsx
Shows predicted breed and confidence as a styled badge, e.g. "Golden Retriever · 91%". Use green for high confidence (>80%), yellow for medium (50–80%), and gray for low (<50%). Helps the user decide whether to accept or override the prediction.
Handle classifier errors gracefully
If the ML server is down, the API times out, or the response is malformed, the app should not crash. Wrap the classifier call in a try/catch. On failure, show a small warning like "Auto-detect unavailable" and let the user type the breed manually. The breed field should already be editable — the classifier just pre-fills it.
Handle the Render cold start delay new
On the free tier, the ML server sleeps after 15 minutes and takes ~30 seconds to wake up. If the first classify request times out, retry once automatically. If it still fails, fall back to manual entry. Optionally ping /health when the report screen opens to wake the server early.
Add breed filtering to sightings.tsx
Now that reports have breed data, add a dropdown filter at the top of the sightings list. Fetch the distinct breeds from your reports with a Supabase query, display them as filter options, and when selected, filter with .eq('breed', selectedBreed). Add an "All Breeds" option to clear the filter.
Add search to the sightings list new
Add a text search bar above the breed filter. Let users search by description text using Supabase's .ilike('description', '%search_term%'). Someone looking for their lost dog might search "brown limping" to find matching sightings. Combine this with the breed filter so both can be active at once.
Add error boundaries and loading states everywhere
Go through every screen systematically and ask: what happens if the network is slow? If Supabase returns an error? If the photo upload fails halfway? If the ML server is unreachable? Each case needs a user-friendly message — never a blank screen, a crash, or a cryptic error. Add a try/catch around every async call and set an error state that renders a red message with a retry button.
Final end-to-end test
Full run-through: sign up with display name → take photo of a dog → breed auto-predicted with confidence badge → override if wrong → add location + description → submit → see report in feed with your name → tap for details → see pin on map with callout → go to sightings → filter by breed → search by description → go to profile → see your reports → delete one → sign out → sign back in → still see remaining reports. Test on both iOS and Android if possible.
Build for distribution with EAS Build
Expo's cloud build service compiles your app without needing a Mac for iOS. Run eas build and distribute via TestFlight (iOS) or direct APK (Android) without going through app stores. You'll need an Expo account and to configure eas.json in your project root — Expo's docs walk through this step by step.
